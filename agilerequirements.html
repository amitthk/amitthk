1.4.2 4 types of tickets in JIRA (developer's desk)

  1. Errors

    You're seing an error page.

  2. Bugs

    Wrong results for right workflow.

  3. Issues

    It works, but result is not desirable one.

    1. Issue type - choose an issue type using these guidelines:
      1. bug - the issue is reporting a defect
      2. epic (product owner) - this will be used by the product owner to create new epics - please do not create issues of this type!
      3. story - the issue is a user story - typically for some larger change or improvement to ONOS
  4. Features Requests

    Something new to be implemented.

1.4.3 3 control points to define prioritization in JIRA (product owner's desk)

  1. Status:
    1. New - this has been reported but not yet reviewed by team for validity
    2. Feedback/Awaiting - We need more information from the person reporting the issue before continuing work
    3. Confirmed - This usually means that someone has verified that the described bug occurs or that the requested feature should be added to the application.
    4. Assigned - A specific person has been named to work on the problem
    5. Resolved - The assignee believes that no more work remains to be done to fix the problem
    6. Closed - Someone, usually the reporter or QA, agrees that the problem as reported has been addressed
  2. Priority: Set this to Normal. Other values should only be assigned after consulting with QA, Tech Support, Professional Services, or Product Management. Here are the values and how we use them:
    1. Urgent - This issue is important enough to delay releasing a new version of the affected software
    2. High - This issue is important and we should address it soon
    3. Normal - Most issues fall into this group
    4. Low - It would be nice to fix this bug or add this feature but it's not particularly important to anyone
  3. Severity: The following list is in order of severity:
    1. Blocker - You cannot perform a business-critical task
    2. Major - You can work but it's difficult
    3. Normal - You can work but the application doesn't behave correctly
    4. Minor - You found a problem but it's either something you don't see very often or easy to work around
